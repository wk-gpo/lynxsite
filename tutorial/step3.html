creating application configuration

You received prototype form UX team. As you played a little bit with it you already get an idea what user expects from that application. You need to establish structure of your project. You will use prototype as starting point in that effort.

1. Add necessary libraries

Of course, the very first library to be used is lynx.js - that's the main reason you read that tutorial, isn't is? But lynx.js is not the only library you need. Lynx need three following libraries: jquery.js, angular.js and store.js. Library angular.js is basis of lynx.js and it requires jquery.js. The last one store.js will enable Local Storage use in web application.

Notice that code samples have hints available. Press button to see all tokens with hints assigned. Move mouse cursor over the token to see hint text. In some case you may also click on token and you will be redirected to page with additional info (e.g. API doc, library site ...).

        As you can see in following code view area following snippets has been extracted:
		

        
login page

        
It will be used in case user authentication is needed. It may be displayed instead of any other application page.

        
home, result list & document view pages

        
Regular application pages as user may see them. Those pages are used by routing mechanism of lynx which we will discuss a little bit later

        
reusable components

        
In our case we found search bar form as to be used on both home and result pages.

    

	

    

        

            

        

        

            

        

        

            

        

        

            

        

        

            

        

        

            

        

    

	
3. Wire-up components in configuration

    

		Lynx application should be properly initialized. You will use to that purpose configuration file which we will name here app.js. That javascript file needs to be loaded in index.html file. You can see on snippet below that it has definitions of application pages. All page routing magic is done by Lynx functionality based on that information.
	

    

        

            

        

        

            

        

	

	

		Configuration you see in app.js is a very basic example but we do not need anything more for now. You can see there is lynx.module function called which configures lynx application. It creates angular.Module object and configures it according to configuration object provided.
	

	

		Configuration parameter moduleName identifies angular.Module lynx creates. You may use that value to find modules by angular.module() function - just use moduleName as parameter.
	

	

		Page structure is constituted in pages parameter. Each object of that array represent one of the pages your application supports. For every page you define:
		

        
name

        
defines internally used page name; it will not appear on URLs

        
path

        
defines structure of anchor which would be used in URL for certain page snippet after page loads

        
view

        
defines relative path base on web application root folder; it points to html snippet file containing page that should be loaded if application is routed to path defined above

	

	

		Application flow is based on switching between defined pages. Routing is based on url hash value. Read more about page Lynx routing in documentation.
	

	

		You should define defaultPath which is used by Lynx application in case routing procedure may not find any other page to display. It should be set to path value of one of your defined pages - usually home page.
	

	

	   

			Notice that login page is not defined in application pages. It should be presented for every url if authentication is required. Login page should not interfere routing of Lynx application.
